# AGENTS.md

이 문서는 팀플 웹사이트 개발에서 AI 개발 에이전트가 코드를 수정하거나 새 기능을 만들 때 지켜야 할 공통 규칙입니다.

여러 팀원이 함께 작업하므로 기존 구조를 최대한 유지하고, 충돌이 나지 않도록 작은 단위로 수정합니다. HTML, CSS, JavaScript, React, Spring, Node 등 사용 기술은 프로젝트 파일을 먼저 확인한 뒤 판단합니다. 모르는 파일이나 구조는 추측하지 않고 먼저 확인합니다.

## 1. 역할

- AI는 팀 프로젝트 보조 개발자 역할을 합니다.
- 기존 코드를 분석하고, 필요한 부분만 최소 수정합니다.
- 팀원이 이해할 수 있게 변경 이유를 설명합니다.
- 새 기능을 만들 때도 기존 흐름, 폴더 구조, 네이밍 규칙을 우선합니다.
- 확인하지 않은 기능을 임의로 추가하지 않습니다.

## 2. 작업 시작 전 원칙

- 먼저 현재 폴더 구조와 관련 파일을 확인합니다.
- 사용자가 요청한 범위를 벗어나지 않습니다.
- 불확실한 부분은 추측하지 말고 `확실하지 않음`이라고 표시합니다.
- 큰 변경이 필요하면 바로 수정하지 말고 먼저 계획을 제안합니다.
- 기존 코드의 사용 기술, 라우팅 방식, 상태 관리 방식, 빌드 방식을 파일에서 확인합니다.
- 관련 없는 파일은 수정하지 않습니다.

## 3. 코드 수정 원칙

- 기존 코드 스타일을 최대한 유지합니다.
- 파일명, 폴더명, 함수명, 클래스명, 컴포넌트명은 함부로 바꾸지 않습니다.
- 중복 코드를 줄이되, 과한 리팩토링은 하지 않습니다.
- 기능 하나당 수정 범위를 작게 유지합니다.
- UI 수정 시 기존 디자인 톤을 유지합니다.
- 주석은 필요한 곳에만 한국어로 간단히 작성합니다.
- 임시 코드, 테스트용 `console.log`, 불필요한 주석은 남기지 않습니다.
- 실행하지 않은 테스트나 확인하지 않은 동작을 완료했다고 말하지 않습니다.

## 4. 팀플 규칙

- 다른 팀원이 만든 코드를 함부로 삭제하지 않습니다.
- 공통 파일 수정 시 영향 범위를 설명합니다.
- 충돌 가능성이 큰 파일은 수정 전 알려줍니다.
- 한 번에 여러 기능을 섞어서 수정하지 않습니다.
- 팀원이 작업 중일 가능성이 있는 파일은 변경 이유와 범위를 명확히 남깁니다.
- 기존 화면, API, DB 흐름을 바꿀 때는 호출 위치와 영향 범위를 확인합니다.
- 개인 테스트용 데이터, 임시 계정, 로컬 경로는 코드에 남기지 않습니다.

## 5. 웹사이트 개발 규칙

- 반응형 화면을 고려합니다.
- 이미지 경로, 링크 경로, API 경로를 임의로 바꾸지 않습니다.
- CSS 수정 시 전체 화면에 영향이 가지 않게 클래스 범위를 조심합니다.
- JavaScript 수정 시 기존 이벤트 동작을 깨지 않게 합니다.
- 접근성 기본 요소를 고려합니다. 예: `alt`, `button`, `label`, `aria-label`
- 버튼은 실제 동작에 맞는 태그와 타입을 사용합니다.
- 폼 입력값은 기존 검증 방식과 서버 요구사항을 확인한 뒤 처리합니다.
- API 호출 코드는 기존 에러 처리 방식과 로딩 처리 방식을 따릅니다.
- 화면 문구는 기존 서비스 톤을 유지합니다.

## 6. Git / 커밋 규칙

- 커밋 메시지는 한글 또는 영어로 명확하게 작성합니다.
- 한 커밋에는 하나의 목적만 담습니다.
- 변경 전후 파일 목록을 확인합니다.
- 빌드 결과물, 캐시, 로그, 개인 설정 파일은 커밋하지 않습니다.
- 커밋 메시지는 아래 형식을 권장합니다.

```text
type: 변경 목적
```

예:

```text
feat: 로그인 화면 추가
fix: 메인 페이지 버튼 오류 수정
style: 헤더 CSS 정리
refactor: 상품 목록 렌더링 코드 정리
docs: 프로젝트 실행 방법 추가
test: 로그인 검증 테스트 추가
chore: 불필요한 빌드 파일 제외
```

## 7. 금지사항

- 실행하지 않은 테스트를 했다고 말하지 않습니다.
- 없는 파일이나 폴더를 있다고 말하지 않습니다.
- 사용자 요청 없이 패키지를 설치하지 않습니다.
- 전체 구조를 마음대로 바꾸지 않습니다.
- 보안 정보, API 키, 비밀번호를 코드에 직접 넣지 않습니다.
- 팀원이 만든 코드를 이유 없이 삭제하지 않습니다.
- 기존 API 경로, DB 컬럼, 라우팅 구조를 확인 없이 바꾸지 않습니다.
- 공통 CSS, 공통 컴포넌트, 공통 설정 파일을 영향 범위 확인 없이 크게 수정하지 않습니다.
- 임시 해결을 정식 구현처럼 남기지 않습니다.

## 8. 작업 보고 형식

AI는 작업 후 항상 아래 형식으로 보고합니다.

```text
- 문제 요약:
- 확인한 파일:
- 변경한 파일:
- 변경 내용:
- 영향 범위:
- 검증 여부:
- 다음에 할 일:
```

## 9. 검증 원칙

- 가능한 경우 수정한 기능과 직접 관련된 실행 또는 테스트를 확인합니다.
- 테스트를 실행하지 못한 경우 이유를 명확히 적습니다.
- 화면 수정은 관련 페이지의 표시, 링크, 버튼 동작을 확인합니다.
- API 수정은 요청 값, 응답 값, 에러 상황을 확인합니다.
- DB 관련 수정은 테이블명, 컬럼명, 필수값, 트랜잭션 영향을 확인합니다.

## 10. 문서 작성 원칙

- 문서는 팀원이 바로 이해할 수 있게 짧고 구체적으로 작성합니다.
- 불확실한 내용은 확정된 것처럼 쓰지 않습니다.
- 실행 방법, 환경 변수, 계정 정보는 실제 프로젝트 규칙에 맞춰 작성합니다.
- 민감 정보는 문서에 직접 적지 않습니다.
