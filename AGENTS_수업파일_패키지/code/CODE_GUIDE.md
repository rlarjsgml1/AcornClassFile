# Code Guide

실제 코드 작업 기준을 언어와 실행 환경별로 나눈 폴더입니다.
코드 작업은 "작게 수정, 명확히 검증, 기존 구조 존중"을 기본으로 합니다.

## 먼저 볼 문서

```text
JAVA_ECLIPSE.md          Java2, Eclipse, 일반 Java 수업 코드
JAVA_WEB.md              JSP, Servlet, Tomcat, WEB_CLOUD
ANDROID_KOTLIN.md        Android Studio, Kotlin, Gradle
JAVASCRIPT.md            JavaScript, 브라우저, Node.js
SQL_DATABASE.md          SQL, DB, DAO와 연결되는 쿼리
CODE_REVIEW_99_CHECKLIST.md 코드 작업 최종 점검표
```

## 공통 확인

- 실제 프로젝트 경로를 먼저 확인합니다.
- 패키지명과 폴더 경로가 맞는지 확인합니다.
- 기존 코드 스타일을 유지합니다.
- 변경 범위가 요청과 직접 연결되는지 확인합니다.
- 컴파일, 실행, 브라우저 확인 등 가능한 검증을 수행합니다.
- 검증하지 못한 항목은 미확인으로 기록합니다.

## 실무형 코드 기준

- 이름은 역할이 드러나게 짓습니다.
- 하드코딩은 필요한 경우에만 사용하고, 반복되는 값은 상수나 설정으로 분리합니다.
- 보안값은 코드에 직접 넣지 않습니다.
- 예외는 숨기지 않습니다.
- 중복 제거는 코드가 더 단순해질 때만 합니다.
- 성능 최적화는 실제 병목이나 요구사항이 있을 때 적용합니다.
- 수업 코드에서는 학습 목적을 우선하고, 실무형 기준은 과하지 않게 적용합니다.

## AI 작업 기준

- 파일 일부만 보고 전체 구조를 단정하지 않습니다.
- 기존 구현이 있으면 새로 만들기보다 재사용 가능성을 먼저 확인합니다.
- 수정 이유가 설명되지 않는 코드는 추가하지 않습니다.
- 테스트를 약화하거나 오류를 숨기는 방향으로 고치지 않습니다.
- 자동 생성 파일과 직접 작성한 파일을 구분합니다.
- 사용자가 요청하지 않은 대규모 리팩터링은 하지 않습니다.

## 의존성 기준

- 새 라이브러리 추가 전 기존 코드와 표준 기능으로 해결 가능한지 확인합니다.
- jar, npm package, Gradle dependency를 추가하면 필요한 이유를 기록합니다.
- lock 파일이 있는 프로젝트는 package manager를 임의로 바꾸지 않습니다.
- 버전 변경은 빌드 영향이 있으므로 사용자 확인 후 진행합니다.

## 변경 전 확인 질문

코드 수정 전 스스로 확인합니다.

- 이 파일이 실제 실행되는 파일인가?
- 같은 역할의 기존 클래스나 함수가 있는가?
- 패키지, import, dependency가 맞는가?
- 사용자가 요청한 범위를 넘지 않는가?
- 검증 방법이 있는가?

## 변경 후 보고 기준

```text
변경:
- 수정한 내용

검증:
- 실행한 명령 또는 확인 방법

주의:
- 미확인 또는 남은 위험
```

## 완료 전 자체 질문

- 이 변경이 사용자 요청과 직접 연결되는가?
- 같은 문제를 더 작게 해결할 방법은 없는가?
- 컴파일, 실행, 화면 확인 중 가능한 검증을 했는가?
- SourceTree에 불필요한 파일이 같이 올라가지 않는가?
