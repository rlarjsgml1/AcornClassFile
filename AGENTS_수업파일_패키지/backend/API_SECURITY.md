# API Security

백엔드 보안, 인증, 입력 검증 기준입니다.

## 1. 기본 보안 원칙

- 사용자가 보낸 값은 신뢰하지 않습니다.
- 화면 검증과 별개로 서버 검증을 반드시 수행합니다.
- SQL은 `PreparedStatement` 또는 바인딩 방식을 우선합니다.
- 비밀번호, 토큰, DB 접속 정보는 코드와 로그에 남기지 않습니다.

## 2. 인증과 권한

- 로그인 여부와 권한 확인 위치를 명확히 합니다.
- 단순 화면 숨김은 보안이 아닙니다.
- 서버에서 권한을 다시 확인합니다.
- 관리자 기능, 수정/삭제 기능은 작성자 또는 권한 기준을 확인합니다.

## 3. 세션 관리

- session key 이름을 통일합니다.
- 로그인 성공 시 필요한 사용자 정보만 세션에 저장합니다.
- 비밀번호는 세션에 저장하지 않습니다.
- 로그아웃 시 session을 정리합니다.
- 세션 만료 시 이동할 화면을 정합니다.

## 4. 입력값 검증

검증 항목:

- 필수값
- 문자열 길이
- 숫자 범위
- 날짜 형식
- enum 또는 선택 가능한 값
- 파일 확장자와 크기
- 페이지 번호와 정렬 기준

## 5. XSS 방지

- 사용자가 입력한 문자열을 JSP에 출력할 때 escape 여부를 확인합니다.
- HTML을 허용해야 하는 경우 허용 태그 정책을 정합니다.
- EL/JSTL 출력 방식과 직접 script 삽입을 구분합니다.

## 6. CSRF 기준

- 로그인 이후 상태를 바꾸는 POST 요청은 CSRF 가능성을 고려합니다.
- 수업 프로젝트에서는 개념 설명을 남기고, 실무 프로젝트에서는 token 방식을 검토합니다.

## 7. API 응답 기준

JSON 응답 예시:

```json
{
  "success": true,
  "message": "처리되었습니다.",
  "data": {}
}
```

오류 응답 예시:

```json
{
  "success": false,
  "message": "입력값을 확인해주세요.",
  "errorCode": "INVALID_INPUT"
}
```

## 8. 민감 정보 기준

로그에 남기지 않을 값:

- 비밀번호
- 인증 토큰
- API key
- DB 접속 정보
- 주민번호, 카드번호 등 개인 민감 정보
